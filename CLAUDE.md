# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Состояние проекта

Это ранний scaffold, а не готовое приложение. В `Assets/` сейчас лежат **только сторонние плагины** (`UniRx`, `UniTask`, `Zenject`) — собственного кода приложения, сцен, ScriptableObject-конфигов и тестов ещё нет. README описывает **целевую** архитектуру, которую предстоит реализовать; раздел «Запуск» там не дописан. При добавлении кода создавай его вне `Assets/Plugins/` (например, в `Assets/Scripts/`), который пока отсутствует.

## Unity и среда

- Точная версия редактора: **2022.3.62f3** (см. `ProjectSettings/ProjectVersion.txt`). Открывай проект именно в ней — Unity жёстко привязывает Library/ к версии.
- Платформа разработки — Windows. Сборочных/линтерных npm-скриптов нет: это чистый Unity-проект, всё происходит внутри редактора.
- `Library/`, `Temp/`, `obj/` — генерируемые, не редактируй и не коммить.

## Работа через MCP for Unity

Проект подключает пакет `com.coplaydev.unity-mcp` (`Packages/manifest.json`). Это основной способ управлять редактором из Claude Code: создание/правка GameObject'ов, сцен, скриптов, запуск тестов, чтение состояния редактора.

**Перед любой работой с Unity-редактором через MCP вызывай скилл `unity-mcp-skill`** — в нём схемы инструментов, паттерны и best practices. MCP требует, чтобы редактор Unity был открыт на этом проекте (MCP-мост работает внутри Editor).

## Тесты

Подключён Unity Test Framework (`com.unity.test-framework` 1.1.33) — Edit Mode и Play Mode тесты. Запускать предпочтительно через Test Runner внутри редактора либо через MCP. Headless-вариант (batchmode), когда редактор недоступен:

```
Unity.exe -runTests -batchmode -projectPath "D:\UnityProjects\unity-request-queue-demo" -testPlatform EditMode -testResults results.xml -logFile -
```

Один тест/фильтр — добавь `-testFilter "Namespace.ClassName.MethodName"`. Для Play Mode замени `-testPlatform PlayMode`.

Собственных тестовых сборок ещё нет; присутствующие `*Tests*.asmdef` принадлежат Zenject и не относятся к коду приложения.

## Целевая архитектура (из README)

При реализации держись этих рамок:

- **Ядро — очередь запросов.** Все сетевые вызовы (`UnityWebRequest`) идут строго последовательно через единую очередь. Очередь должна поддерживать: добавление запроса и получение результата; отмену уже выполняющегося запроса; удаление ещё не стартовавшего запроса. Реализация — на **UniTask + `CancellationToken`** (не на корутинах). Отмена запроса — ключевой сценарий (например, при уходе с вкладки запросы к API отменяются).
- **DI — Zenject.** Зависимости связываются в инсталлерах; экранные модули собираются через DI, а не через ручные `new`/синглтоны.
- **UI — три вкладки с нижней навигацией:** Кликер (валюта по тапу + автосбор, энергия, VFX из ScriptableObject), Погода (периодические запросы с отменой), Породы собак (список + факты через Dog API, попап с адаптивной высотой).
- **Паттерны:** MVC/MVP для экранов, Object Pool для часто создаваемых объектов, конфигурация через **ScriptableObject** (а не хардкод параметров).
- **Реактивность:** UniRx для потоков событий/состояния. README упоминает DOTween для анимаций, но пакет пока не установлен — добавь его перед использованием.

## Сборки (assembly definitions)

Сейчас все `*.asmdef` принадлежат плагинам. Код приложения тоже выноси в собственный `.asmdef`, который ссылается на `Zenject`, `UniTask`, `UniRx` — иначе типы плагинов не будут видны. Не складывай игровой код в существующие сборки плагинов.
